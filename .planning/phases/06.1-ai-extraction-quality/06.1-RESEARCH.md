# Phase 6.1: AI Extraction Quality — Research

**Researched:** 2026-03-21
**Domain:** GPT-4o structured output quality, Zod schema description, schema registry field enrichment
**Confidence:** HIGH (all findings based on direct codebase reading and established GPT structured-output patterns)

---

## Summary

Phase 6.1 targets the single biggest gap in the current workflow: many fields that GPT-4o could plausibly fill are either (a) marked `aiExtractable: false` even though they're visible on build plates or inferable from make/model knowledge, or (b) marked `aiExtractable: true` but given no context beyond their camelCase key name. The result is that the extraction result is thinner than it should be, requiring more manual review than necessary.

The fix has three levers, all in code (no infrastructure changes needed):

1. **Richer `.describe()` strings** in `buildExtractionSchema` — today every field gets an identical generic description. Replacing this with field-specific descriptions (Salesforce label, expected format, where to look in photos, valid options for selects) gives GPT structured output substantially more signal to work with.

2. **An `aiHint` property on `FieldDefinition`** — a single freeform string that captures domain knowledge per field: which plate type it appears on, what prefix/format to expect, and what inference rules apply when the value isn't directly visible. This is the most direct way to encode auction-house domain expertise into the prompt.

3. **More targeted `aiExtractable: true` flags** — several earthmoving, agriculture, forklift, and trailer fields are conservatively marked false even though GPT can reliably extract or infer them. Enabling them (with good hints) directly increases the prefill rate.

**Primary recommendation:** Add `aiHint?: string` to `FieldDefinition`, populate it on every currently-`aiExtractable: true` field and the newly-enabled fields, then update `buildExtractionSchema` to embed the hint in the field's `.describe()` string. Update `buildSystemPrompt` to be more specific about plate types and inference expectations.

## Current State Analysis

### What `buildExtractionSchema` Produces Today

```typescript
// src/lib/ai/extraction-schema.ts — current
shape[fieldKey] = z.object({
  value: z.string().nullable().describe(
    `Extracted value for ${fieldKey}. Return null if not determinable from photos or notes.`
  ),
  confidence: confidenceEnum.describe(
    'high: clearly visible/readable. medium: inferred with uncertainty. low: uncertain guess. null: not found.'
  ),
})
```

Every field receives exactly the same description, parameterised only by its internal snake_case key. GPT-4o receives no information about:
- The human-readable Salesforce field label
- Where in the photos this value typically appears (build plate, compliance plate, instrument cluster, etc.)
- The expected format (e.g., year as 4-digit number, compliance date as MM/YYYY, engine size as "12.7L" vs "12700cc")
- Valid enum values for select fields (Diesel / Petrol / Electric / LPG / CNG)
- Whether this is a direct-read field or a knowledge-inference field

### The `aiExtractable` False-Negative Problem

Cross-referencing all schemas against what GPT-4o can actually do:

**Earthmoving** — only 4 of 34 fields are `aiExtractable: true` (pin, serial, vin, make, model, year, engine_manufacturer). Fields like `fuel_type`, `engine_model`, `horsepower`, `gross_hp`, `emissions_tier`, `transmission`, `drive_type` are all `false`, yet for a known machine (e.g. Caterpillar 320D) these are constants that GPT knows from training data.

**Agriculture** — same pattern: `hourmeter`, `odometer`, `horsepower`, `fuel_type`, `engine_model`, `drive_type`, `transmission` all false.

**Forklift** — `engine_model`, `max_lift_capacity`, `max_lift_height`, `fuel_type`, `colour`, `transmission` all false. A Linde E50 has a known lift capacity.

**Trailer** — only `make`, `model`, `vin`, `year` are true. Yet `colour`, `axle_config`, `suspension`, `tyre_size`, `atm` are all observable from photos or inferable from model.

**Truck** — best coverage, but `cab_type` has no hint about what to look for visually, and `compliance_date` format is unspecified.

### System Prompt Gaps

`buildSystemPrompt` is well-structured but has two gaps:

1. **No plate-type→field mapping** — The prompt says "Read build plates, compliance plates" but doesn't tell GPT which fields live on which plate. A build plate typically contains: Make, Model, Serial/PIN/VIN, Year, Weight Ratings (GVM/GCM/ATM). A compliance plate contains: Compliance Date, Tare. An instrument cluster contains: Odometer, Hourmeter. Being explicit about this mapping prevents GPT from ignoring the build plate entirely when looking for GVM.

2. **Knowledge inference is underspecified** — The example ("a 2011 MACK Granite has known engine options") is good but narrow. The instruction needs to cover the full taxonomy: once make+model+year are identified, infer engine_manufacturer, engine_model, engine_series, horsepower, fuel_type, transmission type, drive_type, suspension type, emissions_tier, axle_configuration, GVM, GCM. For forklifts: max_lift_capacity, max_lift_height, fuel_type. For earthmoving: operating_weight, bucket_capacity, engine specs.

## Architecture Patterns

### Pattern 1: `aiHint` on FieldDefinition

**What:** Optional string on each schema field that encodes extraction guidance for GPT. Embedded into the Zod `.describe()` string by `buildExtractionSchema`.

**Structure of a good hint:**
```
"[Where found]: [plate/source]. [Format]: [expected format]. [Inference]: [what to infer if not visible]."
```

**Examples:**
```typescript
// Build plate / direct-read
{ key: 'vin', label: 'VIN', aiExtractable: true,
  aiHint: 'Found on build plate or compliance plate, 17-character alphanumeric code. Also stamped on chassis rail.' }

{ key: 'compliance_date', label: 'Compliance Date', aiExtractable: true,
  aiHint: 'Found on compliance plate (silver/gold plate, typically inside driver door or under bonnet). Format: MM/YYYY.' }

{ key: 'gvm', label: 'GVM', aiExtractable: true,
  aiHint: 'Found on build plate as "GVM" or "Gross Vehicle Mass". Value in kg, return digits only. Infer from make/model/year if not visible.' }

// Instrument cluster
{ key: 'odometer', label: 'Odometer', aiExtractable: true,
  aiHint: 'Read from instrument cluster display. Digits only, no units. Only extract if clearly readable — do not guess.' }

// Knowledge inference
{ key: 'engine_manufacturer', label: 'Engine Manufacturer', aiExtractable: true,
  aiHint: 'May appear on engine badge or build plate. If not visible, infer from make/model/year (e.g. MACK MP8, Kenworth C15, Volvo D13).' }

{ key: 'fuel_type', label: 'Fuel Type', aiExtractable: true,
  aiHint: 'Valid values: Diesel, Petrol, Electric, LPG, CNG. For heavy trucks and earthmoving assume Diesel unless evidence otherwise.' }

// Select with options
{ key: 'fuel_type', label: 'Fuel Type', inputType: 'select',
  options: ['Diesel', 'Petrol', 'Electric', 'LPG', 'CNG'], aiExtractable: true,
  aiHint: 'Must be exactly one of: Diesel, Petrol, Electric, LPG, CNG. Heavy trucks are almost always Diesel.' }
```

### Pattern 2: Updated `buildExtractionSchema`

The key insight is that the hint + label + options should all be composed into the `.describe()` string:

```typescript
// Pattern — do NOT implement yet, this is research guidance
for (const field of extractableFields) {
  const parts: string[] = []
  parts.push(`Salesforce field: "${field.label}".`)
  if (field.aiHint) parts.push(field.aiHint)
  if (field.options?.length) parts.push(`Valid values: ${field.options.join(', ')}.`)
  parts.push('Return null if not determinable.')

  shape[field.key] = z.object({
    value: z.string().nullable().describe(parts.join(' ')),
    confidence: confidenceEnum.describe(
      '"high" = directly read from photo. "medium" = inferred from vehicle knowledge. "low" = uncertain guess. null = not found.'
    ),
  })
}
```

This requires `getAIExtractableFields` to return `FieldDefinition[]` (not just `string[]`) so hints and options are available. Currently it returns `string[]`. The planner must decide: change the return type of `getAIExtractableFields`, or add a new `getAIExtractableFieldDefs` function. **Recommend adding a new function** to avoid breaking existing callers.

### Pattern 3: Updated `buildSystemPrompt`

Add explicit plate-type → field mapping and expand the inference instruction:

```
// Research guidance — structure of improved system prompt
Step 1 — Identify plates and read them in this priority order:
- BUILD PLATE: contains Make, Model, Serial/PIN/VIN, Year of Manufacture, GVM, GCM, ATM, NW (Nett Weight), Tare
- COMPLIANCE PLATE: contains Compliance Date (MM/YYYY), Tare (kg), ADR numbers
- INSTRUMENT CLUSTER: contains Odometer (km or hours) — only extract if digits are clearly legible
- REGISTRATION PLATE: contains Registration Number
- ENGINE BADGE or VALVE COVER: may show Engine Manufacturer, Engine Series
- WEIGHT RATING PLATE (cab card): GVM, GCM, axle load ratings
- VIN PLATE (chassis): 17-character VIN number

Step 2 — Knowledge inference (once Make + Model + Year are identified):
For TRUCKS: infer engine_manufacturer, engine_series, engine_size, fuel_type, gearbox_make, transmission type, drive_type, suspension, axle_configuration, brakes, GVM, GCM
For EARTHMOVING: infer engine_manufacturer, engine_model, horsepower, fuel_type, drive_type, transmission, emissions_tier
For FORKLIFTS: infer max_lift_capacity, max_lift_height, fuel_type, engine_manufacturer, engine_model
For AGRICULTURE: infer engine_manufacturer, engine_model, horsepower, fuel_type, drive_type, transmission
```

### Anti-Patterns to Avoid

- **Hallucinating VINs and serials** — the system prompt explicitly prohibits this; hints must reinforce it: "Do NOT infer or fabricate serial numbers, VINs, engine numbers, or PIN numbers."
- **Changing `getAIExtractableFields` return type** — it returns `string[]` and is tested by name in `extraction-schema.test.ts`. Add a new function instead.
- **Over-enabling `aiExtractable`** — don't enable fields for which GPT has no training knowledge. Obscure local manufacturers or unusual configurations are better left manual.
- **Putting hints in the system prompt** — hints belong per-field in `.describe()`, not scattered through the prompt. Structured output grounding works best when guidance is co-located with the field.

## Common Pitfalls

### Pitfall 1: Changing `getAIExtractableFields` return type
**What goes wrong:** `extraction-schema.test.ts` and `extract-route.test.ts` both use `getAIExtractableFields('truck')` expecting `string[]`. Changing the return type breaks both test files and any consumer that iterates keys.
**How to avoid:** Add a new `getAIExtractableFieldDefs(assetType): FieldDefinition[]` alongside the existing function. `buildExtractionSchema` calls the new function; everything else calls the old one.

### Pitfall 2: `aiHint` increasing token count past structured-output limit
**What goes wrong:** GPT-4o structured output has a JSON schema size constraint. Very long hint strings across 30+ fields could push the schema description over limits.
**How to avoid:** Keep hints under ~120 characters each. Focus on the single most important piece of information per field. Avoid repeating universal rules in each hint (those belong in the system prompt).

### Pitfall 3: Enabling `aiExtractable` without a hint
**What goes wrong:** The field gets included in the extraction schema but GPT still doesn't know where to look or what to infer. You get nulls but now the field shows up in the review form's low-confidence section unnecessarily.
**How to avoid:** Make it a convention that enabling `aiExtractable: true` requires an `aiHint`. Enforce this in a test: any field with `aiExtractable: true` and `inputType !== 'textarea'` must have `aiHint` defined.

### Pitfall 4: `buildSystemPrompt` not receiving subtype
**What goes wrong:** Currently `buildSystemPrompt(asset.asset_type, asset.asset_subtype ?? '')` passes both. But the subtype-specific inference instructions (e.g. "for Excavators, infer bucket capacity") need the subtype to be meaningful.
**How to avoid:** Use the subtype in the prompt's inference section. "For a ${subtype} (${assetType}), infer..." is more specific than a generic asset type instruction.

### Pitfall 5: Over-inferring during UAT breaks trust
**What goes wrong:** If GPT confidently returns wrong inferred values (e.g. wrong horsepower for a variant), users stop trusting medium-confidence fields.
**How to avoid:** Inference-based fields should default to `confidence: "medium"` per the existing rules. The review form already highlights medium-confidence fields. The risk is contained.

export function getAIExtractableFieldDefs(assetType: AssetType): FieldDefinition[] {

### Updated `buildExtractionSchema` (pattern)
```typescript
// src/lib/ai/extraction-schema.ts — updated pattern
export function buildExtractionSchema(assetType: AssetType) {
  const extractableFields = getAIExtractableFieldDefs(assetType)  // now FieldDefinition[]
  const shape: Record<string, z.ZodTypeAny> = {}
  for (const field of extractableFields) {
    const descParts: string[] = [`Salesforce field: "${field.label}".`]
    if (field.aiHint) descParts.push(field.aiHint)
    if (field.options?.length) descParts.push(`Must be exactly one of: ${field.options.join(', ')}.`)
    descParts.push('Return null if not determinable.')

    shape[field.key] = z.object({
      value: z.string().nullable().describe(descParts.join(' ')),
      confidence: confidenceEnum.describe(
        '"high" = directly visible in photo. "medium" = inferred from vehicle knowledge. "low" = uncertain. null = not found.'
      ),
    })
  }
  return z.object(shape)
}
```
