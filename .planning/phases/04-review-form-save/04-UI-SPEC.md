## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, badge-to-label gap, inline padding |
| sm | 8px | Between label and input within a field row; gap between checklist action buttons |
| md | 16px | Default element spacing; field row vertical padding (py-4); card inner padding |
| lg | 24px | Section spacing between form and checklist; page horizontal padding |
| xl | 32px | Major section top margin; sticky CTA bottom padding on mobile |
| 2xl | 48px | Page top padding |
| 3xl | 64px | Not used in this phase |

Exceptions: Touch targets for checklist action buttons (Dismiss / Mark Unknown / Confirm) must be minimum 44px tall on mobile. Apply `min-h-[44px]` to these buttons.

Source: 8-point scale standard; existing `ExtractionResultPanel` uses `py-3` (12px) per row — Phase 4 review form uses `py-4` (16px) to accommodate the editable input widget in each row.

### Semantic Confidence Colors

These are additive to the 60/30/10 split — they come from the existing `ConfidenceBadge` component and must not change:

| Confidence Level | Color | Tailwind Class | Usage |
|-----------------|-------|----------------|-------|
| high | green-400 | `text-green-400` | Badge text + icon; no field row highlight |
| medium | amber-400 | `text-amber-400` | Badge text + icon; field row gets `border-l-2 border-l-amber-400/40 pl-3` |
| low | white/40 | `text-white/40` | Badge text + icon; field row gets `border-l-2 border-l-red-500/40 pl-3` |
| not_found | white/40 | `text-white/40` | Badge text + icon; field row gets `border-l-2 border-l-red-500/40 pl-3` — same treatment as low |

Design decision (Claude's Discretion — resolved): Medium confidence uses a subtle amber left border (`border-l-amber-400/40` at 40% opacity). Low and not_found use a red left border (`border-l-red-500/40` at 40% opacity). Both use `pl-3` (12px) indent to visually group label + input under the border. High confidence has no border highlight — the green badge alone is sufficient feedback.

Source: CONTEXT.md — "must be distinguishable without being alarming for medium confidence". Amber at 40% opacity achieves caution without alarm. Red at 40% opacity signals attention without being visually loud.

## Interaction Contracts

### Field Row States

Each field row in `DynamicFieldForm` has exactly four visual states:

| State | Trigger | Visual Treatment |
|-------|---------|-----------------|
| High confidence | `confidence === 'high'` | No border highlight; green ConfidenceBadge; input pre-filled |
| Medium confidence | `confidence === 'medium'` | `border-l-2 border-l-amber-400/40 pl-3`; amber ConfidenceBadge; input pre-filled |
| Low / not found | `confidence === 'low'` or `'not_found'` | `border-l-2 border-l-red-500/40 pl-3`; white/40 ConfidenceBadge; input pre-filled or empty |
| Focused (any) | Input focused | `ring-2 ring-[var(--ring)]` on input element via shadcn default |

### Checklist Item States

| State | Trigger | Visual Treatment |
|-------|---------|-----------------|
| Flagged (blocking) | Unresolved, `isBlocking: true` | "Required" badge in red-ish; field name; no dismiss action |
| Flagged (dismissible) | Unresolved, `isBlocking: false` | "Optional" badge in muted; field name; "Not applicable" button shown |
| Confirmed | User typed a value into the field | Green CheckCircle2 icon; field name; value shown |
| Unknown | User pressed "Unknown / not available" | MinusCircle icon; field name; "unknown" text shown |
| Dismissed NA | User pressed "Not applicable" | MinusCircle icon; field name greyed out; "Not applicable" shown |

### Save Button States

| State | Condition | Visual Treatment |
|-------|-----------|-----------------|
| Active | All blocking checklist items resolved | `bg-[var(--primary)] text-white` — full opacity |
| Disabled | Any blocking item still `flagged` | `opacity-40 cursor-not-allowed`; "Resolve required fields before saving." copy above button |

### Re-Extraction Conflict Banner

Shown only when re-extraction returns values that differ from staff-edited (dirty) fields. Banner appears above the field list, not inline. Two buttons: "Use AI Values" and "Keep My Edits". Banner auto-dismisses on either action. If no dirty field conflicts, re-extraction silently overwrites clean fields.

### Sticky CTA Pattern

The Save button follows the same sticky-bottom pattern as `ExtractionResultPanel`:
- `sticky bottom-0 bg-[var(--background)] pt-3`
- Full-width, height `h-11` (44px — meets touch target minimum)
- "Re-run Extraction" as secondary text link below the primary CTA button (matches existing pattern from `ExtractionResultPanel.tsx` line 68)

Source: `ExtractionResultPanel.tsx` — direct pattern reuse for visual consistency.

## Layout Contract

| Property | Value |
|----------|-------|
| Route | `/assets/[id]/review` |
| Page structure | Server Component wrapper → `ReviewPageClient` (client) |
| Form layout | Flat scrollable list, fields in `sfOrder`. No sections, no pagination. |
| Checklist placement | Below the field list, above the sticky CTA. Separated by `<Separator>`. Always visible — not a modal or drawer gate. |
| Mobile scroll | Single column; form + checklist + CTA in one vertical scroll |
| Desktop max-width | `max-w-2xl mx-auto` — narrow column matching extraction panel |

Design decision (Claude's Discretion — resolved): Checklist placed below form (not above, not in modal). Staff complete the form first to fill gaps, then the checklist shrinks naturally as fields are entered. A modal gate would interrupt mid-edit flow. An above-form checklist distracts before staff have had a chance to edit.
