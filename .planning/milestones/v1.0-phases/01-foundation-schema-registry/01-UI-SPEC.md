## Spacing Scale

Declared values (multiples of 4 only):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline label padding |
| sm | 8px | Between form field and its label, compact button padding |
| md | 16px | Default padding inside cards, form group spacing |
| lg | 24px | Section padding within a screen, between card groups |
| xl | 32px | Between major screen sections (e.g. header to content) |
| 2xl | 48px | Page-level vertical rhythm, bottom safe-area padding on mobile |
| 3xl | 64px | Not used in Phase 1 — reserved for future layout phases |

Exceptions:
- Touch targets for all interactive controls: minimum 44px height (iOS HIG compliance — staff use phones on-site)
- Branch selector chips and asset type cards: minimum 48px height so they are tap-friendly in portrait

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px | 400 (regular) | 1.5 |
| Label | 14px | 400 (regular) | 1.4 |
| Heading | 20px | 600 (semibold) | 1.2 |
| Display | 28px | 600 (semibold) | 1.15 |

Notes:
- Body (16px/400) is the default for all paragraph text, form input values, and list items
- Label (14px/400) is used for form field labels, helper text, and secondary metadata
- Heading (20px/600) is used for screen titles and section headers within a view
- Display (28px/600) is used only for the "Slattery Auctions" branded text on the login page (logo fallback)
- Font stack: Inter via `next/font/google` — no external CDN calls at runtime

## Color

Brand palette: dark green backgrounds, white text, dark navy as secondary surface. Design is dark-mode by default.

| Role | Value | Usage |
|------|-------|-------|
| Primary background | #166534 | Page backgrounds, card surfaces, main content areas |
| Secondary / elevated | #14532D | Nav bar, step headers, elevated cards, bottom navigation |
| Navy accent | #1E3A5F | Borders, active state indicators, bottom nav background |
| Text (primary) | #FFFFFF | All headings, body text, labels — white on dark surfaces |
| Text (muted) | rgba(255,255,255,0.65) | Secondary labels, helper text, placeholder text |
| Destructive | #F87171 | Error states — lighter red readable on dark green backgrounds |

Interactive element highlights:
1. "Sign In" button — navy (#1E3A5F) background, white text
2. "New Asset" button / FAB — navy (#1E3A5F) background, white text
3. "Next" / "Continue" button — navy (#1E3A5F) background, white text
4. Active/selected card — white border (2px) + subtle white overlay (rgba(255,255,255,0.1))
5. Selected branch chip — white border (2px), white text

Logo area: "Slattery Auctions" in white (#FFFFFF) on dark green (#166534) background — no separate container needed, inherits page background.

Dark surfaces are the default. No light-mode fallback required for Phase 1.

## Layout Contract

### Mobile-first portrait layout (primary)
- Viewport: 390px wide (iPhone 15 base) as design target
- All screens use full-width single-column layout in portrait
- No horizontal scrolling permitted
- Safe area insets respected: `env(safe-area-inset-bottom)` for bottom padding
- Navigation: top app bar (title + optional back/action icons) + bottom tab bar or equivalent bottom-anchored FAB

### Desktop (secondary — same functionality, wider content column)
- Content column max-width: 480px centered for form-heavy screens (login, new asset flow)
- Asset list: max-width 640px centered on desktop

### Component patterns
- Login page: vertically centered card (max-width 400px), "Slattery Auctions" display text at top, email input, password input, Sign In button — no decorative imagery
- Asset type selection: 2-column grid of cards in portrait (3-column on desktop), each card has a lucide-react icon (48px) + asset type label
- Branch selection: horizontally scrollable chip row OR single-column scrollable list — executor discretion; chips preferred if they fit without horizontal overflow at 390px

## Interaction Contract

### Login flow
1. User enters email + password
2. Taps "Sign In"
3. Button enters loading state (spinner replaces label, button disabled) — no delay before showing loading
4. On success: redirect to `/` (asset list)
5. On error: inline error message below the form (not a toast) — form inputs remain populated, button re-enables

### New Asset entry flow
1. User taps "New Asset" → navigates to `/assets/new` (step 1: branch)
2. Last-used branch is pre-selected if available (highlight with accent color)
3. User taps a branch → selection is visually confirmed (accent border/background) → "Next" button activates
4. User taps "Next" → step 2: asset type grid
5. User taps an asset type card → card shows selected state (accent border + checkmark icon) → "Next" activates
6. User taps "Next" → step 3: subtype list for selected type
7. User taps a subtype → selection confirmed → "Next" / "Continue" activates
8. User taps "Continue" → asset record created → proceed to Phase 2 (photo capture) — Phase 2 is out of scope here, so this step creates the record and lands on a placeholder or photo capture stub

Back button on steps 2 and 3 returns to the previous step without losing prior selections.

### Loading states
- Auth session check on page load: show a centered spinner (lucide-react `Loader2` icon, animated, white) on the primary green (#166534) background — do not flash the login page before the session check completes
- "Sign In" button loading: replace button text with `Loader2` spinner, disable button, maintain button width
- No skeleton loaders needed in Phase 1 (asset list is empty on first load; no data-dependent skeletons required)

### Error states
- Auth errors: inline below the form, 14px/400 text in destructive red (#F87171 — visible on dark green background), no icon needed
- Network errors: same inline placement
