## Cross-Phase Integration

The key integration path for v1.4:

```
schema file subtypes (Phase 16)
  → SCHEMA_REGISTRY.getSubtypes()
  → wizard Step 3 AssetSubtypeSelector
  → createAsset() DB write (asset_subtype: snake_case_key)
  → /api/describe route reads asset_subtype
  → buildDescriptionUserPrompt injects "Subtype: {key}"
  → GPT-4o matches key against DESCRIPTION_SYSTEM_PROMPT headings (Phase 17)
  → structured description output
```

**Fully wired:** SUBTYPE-01 through SUBTYPE-08, DESCR-01 through DESCR-03, DESCR-05, DESCR-07 (11 of 16 requirements have clean end-to-end paths).

**Partially wired (non-blocking, semantic inference required):**

| Gap | Requirement | Details |
|-----|-------------|---------|
| `washing` key → `WASHING PLANT` heading | DESCR-04 | Schema key and prompt heading differ; GPT-4o must bridge; test uses phantom key `washing_plant` |
| `ewp` forklift key → `EWP (FORKLIFT-MOUNTED)` heading | DESCR-06 | Ambiguous against truck `EWP (ELEVATED WORK PLATFORM)`; asset_type context disambiguates but untested with real key |
| `stock_picker` → `STOCK PICKER / ORDER PICKER` heading | DESCR-06 | Test uses phantom key `order_picker`; real key untested |
| `private`, `recreational` marine keys | DESCR-08 | No dedicated sections; fall through to `MARINE (RECREATIONAL BOAT)` / `OTHER MARINE VESSEL` by inference |

**Note:** The describe API relies entirely on GPT-4o semantic inference to map snake_case subtype keys to UPPERCASE prompt headings. No code-level mapping table exists. For most subtypes this is a reasonable design given the AI context, but the earthmoving `washing`/`WASHING PLANT` mismatch and the marine `private`/`recreational` gap represent the weakest links.
